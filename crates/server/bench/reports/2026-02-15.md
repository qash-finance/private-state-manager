# Benchmark Report

## Introduction
This report summarizes local benchmark runs for the PSM server using the benchmark harness in `crates/server/bench`.

Summary of included test groups:
1. Filesystem vs Postgres baseline comparison (`state-read`, `state-write`, `mixed`).
2. Postgres state-sync with `1000` users (`4 get_state : 1 push_delta`).
3. Postgres auth-scheme comparison (Falcon vs ECDSA).
4. Postgres canonicalization under concurrent load.
5. Filesystem vs Postgres higher-load comparison.
6. HTTP vs gRPC using `configure`, `get_state`, and `push_delta` flow.
7. Postgres stress runs at `10000` users (`state-sync` and `state-read`).

## 4) Canonicalization (Postgres)

### Goal
Measure end-to-end canonicalization behavior under concurrent users:
- submit deltas (`push_delta`)
- wait until each delta reaches terminal state (`canonical_at` or `discarded_at`)

### What this benchmark actually executes
- Seeding phase (not included in measured scenario latency):
  - `configure` for each account.
- Measured canonicalization phase:
  - `push_delta` once per account operation.
  - poll `get_delta` until terminal status or timeout.

### Run context
- Main run dir: `crates/server/bench/results/20260213_192305_postgres`
- Sanity run dir: `crates/server/bench/results/20260213_192052_postgres`

### Main workload configuration
- `users=100`
- `accounts=1200`
- `signers_per_account=2`
- `scenario=canonicalization`
- canonicalization poll interval: `250ms`
- canonicalization timeout: `180s`

### Concurrency model
- The loadgen starts one worker per user, so `100` workers run in parallel.
- In canonicalization scenario, each worker executes one account-operation at a time (sequential per worker).
- With `1200` accounts over `100` users, each worker handles about `12` canonicalization operations.
- Approximate max in-flight canonicalization workflows is bounded by user count (`~100`).
- Polling pressure upper bound at peak is about `100 * (1000/250) = ~400 get_delta req/s`.

### Main results (`20260213_192305_postgres`)
- `total_ops=1200`
- `success_ops=1198`
- `failed_ops=2`
- `success_ops_per_sec=22.11`
- `p50_ms=2968.81`
- `p95_ms=13983.85`
- `p99_ms=17268.77`
- `max_ms=18908.37`

Timing context:
- Seeding: `674063.91 ms` (`11.23 min`)
- Measured canonicalization phase: `54179.64 ms` (`54.18 s`)

### Failure interpretation
- The 2 failures were:
  - `Authentication failed: Request timestamp outside allowed window ... (max 300000ms)`
- These are benchmark operational/config timing failures (request timestamp window), not canonicalization correctness failures.
- Interpretation: long or queued requests can cross the auth window. This should be treated as benchmark or auth-window tuning, not protocol failure.

### Sanity run results (`20260213_192052_postgres`)
- `users=2`, `accounts=8`
- `total_ops=8`, `success_ops=8`, `failed_ops=0`
- `success_ops_per_sec=1.02`
- `p95_ms=2150.15`

### Observations
- Canonicalization path works under parallel load and reaches terminal states in the large run.
- Tail latency is materially higher than non-canonicalization scenarios (`p95 ~14s`, `p99 ~17s`), so canonicalization is a key latency bottleneck.
- This benchmark is meaningful for concurrent progression to terminal delta status. It is not just a simple availability check.

- `p50_ms=263420.79`
- `p95_ms=517443.07`
- `p99_ms=539222.81`
- `max_ms=545176.26`

Timing:
- Seeding: `1240670.77 ms` (`20.68 min`)
- Measured run: `545398.73 ms` (`9.09 min`)
- Combined loadgen time: `29.77 min`

Resource snapshot:
- `cpu_avg=36.98%`
- `cpu_max=189.40%`
- `rss_max_mb=671.34`

### Observations
- Postgres handles 10k-user setup and request volume, but tail latency is extremely high under this local-machine profile.
- `state-sync` at 10k users exceeded auth timestamp tolerance for many requests, causing a `14.1%` failure rate (`7053/50000`).
- `state-read` at 10k users with only one op per user completed with `0` failures, but with multi-minute latency tails.

### Why delays happened at 10k users (likely causes)
- The auth timestamp acceptance window is `5 minutes`, while measured tail latency exceeded that in the 10k state-sync run (`p95 ~7.15 min`), which explains the timestamp-expiry failures.
- Postgres concurrency is capped at `16` connections in metadata and `16` in storage, which is low for a `10k` concurrent-user fan-out and can create large queues.
- Every authenticated request performs metadata read + replay-protection CAS update of `last_auth_timestamp`, adding DB roundtrips and write contention.
- `push_delta` includes a serialized critical section around `network_client.lock()`, which can bottleneck write/sync-heavy workloads.

Code locations for these bottlenecks:
- Auth skew check (`MAX_TIMESTAMP_SKEW_MS`, request timestamp validation):
  - `crates/server/src/metadata/auth/credentials.rs`
  - `crates/server/src/services/mod.rs`
- Postgres pool sizes (`max_size(16)`):
  - `crates/server/src/metadata/postgres.rs`
  - `crates/server/src/storage/postgres.rs`
- Replay-protection CAS update path:
  - `crates/server/src/metadata/postgres.rs`
- Serialized lock in delta apply path:
  - `crates/server/src/services/push_delta.rs`
